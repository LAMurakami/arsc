HPCMP BCT Common Set of Environment Variables policy

  • The following statement is repeated in the ARCHIVE_HOST and ARCHIVE_HOME sections in HPCMP BCT Common Set of Environment Variables policy should be corrected or removed:
    • Note this environment variable becomes unnecessary and should be undefined once Storage Lifecycle Management (SLM) goes into production at a center.
  • The repeated statement in the ARCHIVE_HOST and ARCHIVE_HOME sections should/could read:
    • Note the definition and use of this environment variable will change once Storage Lifecycle Management (SLM) goes into production at a center.
    • The combination of the unique values of ARCHIVE_HOME and ARCHIVE_HOST can be used to determine the default resource and location of an individual user's directory on the archival file system that serves a given compute platform. Although the access means will change each individual user will still have a directory on the archival file system that serves a given compute platform.
  • On 12/16/2013 Lawrence stated that we want Don Cable to take this to the BCT.
    • Now that we have the slm module using the ARCHIVE_HOME environment variable we show a clear need for a change of the "this environment variable becomes unnecessary" statement.
  • A module to "properly" set ARCHIVE_HOME at NAVY was created and installed ant NAVY to be loaded prior to the slm module.
    • NAVY sets ARCHIVE_HOME to HOME (/u/home/${USER}) for all users by default.
    • Setting ARCHIVE_HOME to /u/g/${USER} or /u/b/${USER} appropriately does not interfere with with any use of the ARCHIVE_HOME environment variable since users can use either the automounted HOME or the actual file-system directory.
  • On 1/13/2014 Don acknowledged that Lawrence wants the "this environment variable becomes unnecessary" statement removed from the HPCMP BCT Common Set of Environment Variables policy.
  • On 1/27/2014 and 2/3/2014 Don reported no status change on this item.
  • On 3/24/2014 Don stated that he will talk to the BCT about this.
